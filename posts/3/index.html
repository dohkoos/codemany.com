在Web应用中保存Session数据

Jan 30th, 2013

保存在客户端 Cookie是一个很好的机制。有时候还可以使用隐藏的表单字段（hidden form fields）。

保存在服务器上 最自然的选择。问题是当不止一台前端服务器时，需要能支持Session复制（session replication）和故障转移（failover）的集群。不管哪个前端服务器收到请求，Session数据都在上面。问题是每次请求时可能都要有数据复制和故障转移的动作，影响性能。有个更简单的方法是Sticky sessions。当客户第一次请求时，负载均衡服务器分配一个Cookie给客户，下次请求时就可以通过Cookie访问与上次相同的服务器，这样客户的Session数据就只要在单台服务器上。Sticky sessions也有两个问题：单点故障和负载平衡可能不是最优。

保存在数据库中 通常数据库成为了性能瓶颈，解决方法是花费大笔的金钱购买Oracle或IBM的数据库集群。

使用独立的Session存储 可以考虑Memcached，当然也可以使用Oracle Coherence。

使用Struts 2 Validation框架校验登录信息

Sep 8th, 2012

Struts 2有三种校验方式：
1. 使用Annotation进行校验
2. 使用XML配置校验
3. 使用覆盖validate方法校验

使用XML配置时validation.xml的命名规则和放置路径： -validation.xml

就是要验证的Action类的名字，将此文件放于与Action类文件相同的目录下。

如果该Action类在struts.xml配置中有多个action实例，那么对应某个action实例的验证文件名规则如下： –-validation.xml

例如：UserAction-login-validation.xml。注意，这里的并不是方法名，而是struts.xml中配置的action实例的名字。

既然用框架来校验了，那么在LoginAction.java中判断用户名和密码是否为空的代码就可以删除了。还要记得加上username和passowrd的getter方法，不然校验器（validator）会因为取不到field而报错。

同时在LoginAction.java相同目录下创建LoginAction-validation.xml，内容如下：
[code lang=“xml”] <!DOCTYPE validators PUBLIC

"-//OpenSymphony Group//XWork Validator 1.0.2//EN"<br />
"http://www.opensymphony.com/xwork/xwork-validator-1.0.2.dtd">


<field name="username">
    <field-validator type="requiredstring">
        <message>Username is required</message>
    </field-validator>
</field>
<field name="password">
    <field-validator type="requiredstring">
        <message>Password is required</message>
    </field-validator>
</field>

“`

message提供了校验出错的信息，消息中可以使用${}来引用被校验的对象。如果未出现异常，则转入用户请求的处理方法，如果出现异常，则转入input所指定的视图。所以，对于验证的action实例，必须要在配置文件中为其指定input视图。

package com.codemany.account.dao.hibernate;

import org.hibernate.Session;
import org.hibernate.SessionFactory;
import org.hibernate.cfg.Configuration;

public class HibernateUtil {

private static final SessionFactory sessionFactory;

static {

    try {<br />
        // Create the SessionFactory from hibernate.cfg.xml<br />
        Configuration configuration = new Configuration();<br />
        configuration.configure();<br />
        Properties props = configuration.getProperties();

ServiceRegistry serviceRegistry =

                new ServiceRegistryBuilder().applySettings(props).buildServiceRegistry();

sessionFactory = configuration.buildSessionFactory(serviceRegistry);

    } catch (Throwable ex) {<br />
        // Make sure you log the exception, as it might be swallowed<br />
        System.err.println("Initial SessionFactory creation failed." + ex);<br />
        throw new ExceptionInInitializerError(ex);<br />
    }<br />
}

public static Session getSession() {

    return sessionFactory.openSession();<br />
}<br />

}
